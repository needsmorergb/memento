<objective>
Deliver the host-facing review experience on both clients: a "Review same-day edit" card that plays the compiled video and offers Approve (high-consequence confirm → fans out push + email) and Regenerate (lighter confirm), reusing the existing video-status polling. Critically, the mobile host card is moved OFF the public `useGetEventVideoStatusByToken` onto the authed `useGetEventVideoStatus` so the unapproved review cut is never served on the public path. This is the final vertical slice — after it, a host can preview-and-approve end-to-end (VIDEO-01 + VIDEO-02). Wave 3, after the endpoints (Plan 02) and generated hooks (Plan 01) exist.

Purpose: Make the review gate usable by a real host — the backend gate is invisible until there is a player and Approve/Regenerate UI.
Output: Edited web `event-detail.tsx` + mobile `event.tsx`; one human-verify checkpoint covering the visual/interaction contract on both clients.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile host card → video-status source | Host card must read the authed endpoint, never the public token endpoint (which withholds the unapproved cut). |
| host UI → approve mutation | Approve triggers an irreversible all-guests notification; the UI must require an explicit, informed confirm. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | mobile host review card | mitigate | Host card switched from `useGetEventVideoStatusByToken` (public) to authed `useGetEventVideoStatus`; the public path (Plan 02) withholds videoUrl pre-approval so even a mis-render cannot leak the cut. Verified in checkpoint step 4. |
| T-04-02 | Repudiation / abuse of trust | Approve action | mitigate | Approve is behind a confirm that explicitly states push + email + all-guests + un-sendable; reduces accidental premature delivery. Idempotency enforced server-side (Plan 02). |
| T-04-03 | Tampering | client-side gating | accept | Client only renders actions for ready_for_review; the authoritative host-only + state checks are server-side (Plan 02 approve/regenerate 404/409). Client UI gating is convenience, not the security boundary. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages (expo-av already installed; RESEARCH says do NOT add expo-video this phase). No install task. |
</threat_model>
