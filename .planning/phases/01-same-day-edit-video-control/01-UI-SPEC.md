# Phase 1 — UI Design Contract

> Visual and interaction contract for the host "Review video" experience on both clients (memento-web + memento-mobile). Brownfield: reuse the existing event-detail surface, tokens, and components — do NOT introduce a new design system. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> Scope: the host previews the compiled same-day-edit video and either **Approves** (fans out push + email to guests) or **Regenerates** it. Guests see nothing until approval. Themes/music and guest-facing video changes are deferred (VIDEO-04/05).

---

## Design System

Detected from the codebase — both clients already share a warm Linen / Espresso / Rust palette and Outfit/Playfair fonts. The contract below is the existing system, not a new one.

| Property | Value |
|----------|-------|
| Tool | shadcn (web only) |
| Preset | `style: new-york`, `baseColor: neutral`, `cssVariables: true`, `rsc: false`, `tsx: true` (`artifacts/memento-web/components.json`) — tokens overridden by the project theme in `src/index.css`, NOT stock neutral |
| Component library | Web: Radix UI via shadcn/ui (`@/components/ui/*`, ~50 primitives present). Mobile: no component library — hand-built React Native primitives styled via `StyleSheet` + `useColors()` |
| Icon library | Web: `lucide-react`. Mobile: `@expo/vector-icons` (`Feather`) |
| Font | Sans: **Outfit** (body/UI). Serif: **Playfair Display** (headings, web `font-serif`). Mono: Menlo. Mobile loads `Outfit_400Regular / _600SemiBold / _700Bold` (the contract uses 400 + 600, with 700 reserved for the serif display heading only — see Typography) |
| Video player | Web: native `<video controls>` pointed at the signed playback URL. Mobile: `expo-av` `Video` (`useNativeControls`, `ResizeMode.CONTAIN`) — already the pattern in `app/video.tsx` |
| Polling | Reuse existing React Query video-status hooks. **Web host:** `useGetEventVideoStatus` (authed). **Mobile host:** must switch the host's review card to the authed `useGetEventVideoStatus` hook — `app/(tabs)/event.tsx` currently uses the public `useGetEventVideoStatusByToken`, which MUST NOT surface the review video before approval |

## Spacing Scale

The codebase uses Tailwind's default 4px-based scale on web (`gap-3` = 12, `p-6` = 24, `py-8` = 32) and explicit pixel values on mobile (`StyleSheet`). Declared contract (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge inner padding |
| sm | 8px | Compact stacking (button rows, pill gaps) — web `gap-2`, mobile `gap: 8` |
| md | 12px | Card internal element spacing — web `gap-3`, mobile `gap: 12` (the dominant in-card rhythm) |
| md+ | 16px | Default block spacing, card body padding — web `p-4`/`mb-6` region, mobile `padding: 16` / `gap: 16` |
| lg | 24px | Card padding on web (`p-6`), section padding |
| xl | 32px | Main content vertical padding (`py-8`) |
| 2xl | 48px | Major empty-state vertical padding (web uses `py-16`/`py-20` for full-card empty states) |

Exceptions:
- Empty/processing state cards on web use `py-16`–`py-20` (64–80px) to center hero messaging — keep this for the "compiling" and "regenerating" full-card states.
- Mobile touch targets: action buttons (`Approve`, `Regenerate`, `Watch`) keep a **minimum 46px height** (existing `watchBtn`/`signInBtn` convention), and the back/share icon bubbles are 36px — preserve these, do not shrink.
- **Accepted brownfield tokens:** 12px (`gap-3`) and 80px (`py-20`) fall outside the canonical 8-point set but remain 4px-grid-aligned; they are inherited from the existing event-detail surface and are accepted as-is rather than refactored in this phase.

## Typography

Web roles map to existing classes; mobile maps to existing `StyleSheet` font sizes + Outfit weights. The contract uses exactly **two weights**: regular **400** (body, label, meta) and semibold **600** (section headings, card titles, status verbs). The only exception is the single largest serif **display heading**, which retains Playfair **700** as a brand-required treatment — no other element on the surface uses 700, and 500 (medium) is not used at all.

| Role | Web | Mobile | Weight | Line Height |
|------|-----|--------|--------|-------------|
| Display / hero heading | Playfair `font-serif text-2xl font-bold` (24px) | `eventTitle` 26px | 700 (serif — sole brand exception) | 1.2–1.25 (web `tight`, mobile `lineHeight: 32`) |
| Section heading | `font-semibold` ~16px (card `h3`) | `sectionLabel` 16px | 600 | 1.3 |
| Body | `text-sm` (14px) / base 16px | `videoSub` / `infoValue` 15px | 400 | 1.5 (mobile `lineHeight: 18` on 13px sub-text) |
| Label / meta | `text-xs` (12px) | `infoLabel` / `pillText` 12–13px | 400 | 1.4 |

Rules:
- Two-weight rule: every element is **400** or **600**. The serif display heading at **700** is the single permitted exception. **500 (medium) is not used anywhere** on this surface.
- The card title for the review section is the existing pattern: `font-semibold` (600) with a leading 16px icon (`Film` web / `Feather "film"` mobile) tinted `text-primary`.
- Status verbs ("Ready for review", "Compiling…", "Regenerating…", "Approved & delivered", "Failed") use semibold weight **600**, never 700/500 and never the serif display face.
- Do not introduce new font families or sizes beyond those listed — both clients already constrain to Outfit + Playfair.

## Color

Detected theme (web `src/index.css` `:root`; mobile `constants/colors.ts`). Identical intent across clients. The 60/30/10 split below describes the review surface.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` `hsl(36 33% 96%)` ≈ `#faf7f2` (warm linen) | Page background, surfaces |
| Secondary (30%) | `--card` `hsl(36 40% 98%)` ≈ `#fdfcfa` + `--muted` `hsl(30 20% 90%)` ≈ `#ebe6df` | Review card surface, video letterbox is `bg-black`, muted chips/sub-rows |
| Accent (10%) | `--primary` Rust/Terracotta `hsl(13 49% 48%)` ≈ `#b05c3c` (mobile `#b05c3c`) | See reserved-for list below |
| Destructive | `--destructive` `hsl(0 70% 50%)` ≈ `#cc2929` | Regenerate confirm emphasis + failed-state text/icon only |

Accent (Rust/Terracotta) reserved for:
- The **primary "Approve & notify guests" CTA** (solid `Button` default variant on web, solid primary button on mobile — the highest-emphasis action on the surface).
- The active processing spinner / status icon (`Loader2` web, `ActivityIndicator` mobile) and the `Film` section-heading icon.
- The "ready for review" status accent (icon + small label), so the host's eye lands on the action.

Accent is NOT used for: Regenerate (secondary `outline` variant), Download, back/navigation, body text, or card borders.

State-color guidance (reuse existing inline patterns, do not invent new tokens):
- **Ready for review:** neutral card + primary accent on the status row and the Approve CTA.
- **Processing / compiling / regenerating:** `text-muted-foreground` body with a primary-tinted spinner (existing `Loader2 animate-spin text-primary`).
- **Approved & delivered:** the existing green success treatment already in the codebase (`text-green-600` web `CheckCircle`; mobile `#16a34a` / `#f0fdf4` card from `event.tsx`) — surface as a confirmation, with both action buttons removed/disabled.
- **Failed:** `text-destructive` + `AlertCircle`, on web inside `border-destructive/20 bg-destructive/5`; offer Regenerate as the recovery action.

## Copywriting Contract

Tone matches existing copy ("Same-day edit", "Editing your memories…", warm and plain). Approve is high-consequence (notifies every guest) — copy must make the irreversibility-of-notification explicit.

| Element | Copy |
|---------|------|
| Section heading | `Review same-day edit` (web `h3` + `Film` icon; mobile `videoCardTitle`) |
| Ready status line | `Ready for your review — guests haven't been notified yet` |
| Primary CTA | `Approve & notify guests` |
| Secondary CTA | `Regenerate edit` |
| Processing state heading | `Compiling your same-day edit…` |
| Processing state body | `We're stitching the photos, videos, and voice notes together. This usually takes a few minutes — you'll review it here before any guest sees it.` |
| Regenerating state heading | `Building a fresh edit…` |
| Regenerating state body | `Your previous edit was discarded. The new version will appear here for review. Guests still haven't been notified.` |
| Empty state heading (ended, no job yet) | `No edit yet` |
| Empty state body | `The same-day edit starts compiling right after the event ends. Check back in a moment.` |
| Approved state heading | `Approved & delivered` |
| Approved state body | `Guests have been notified by push and email. They can now watch the edit.` |
| Error state (failed) | `We couldn't finish the edit. {errorMessage}` + recovery: `Regenerate edit` to try again. |
| Destructive confirmation — Approve | Title: `Send this edit to all guests?` Body: `Approving notifies every guest by push and email, and they'll be able to watch it. You can't un-send a notification.` Confirm: `Approve & notify` / Cancel: `Keep reviewing` |
| Destructive confirmation — Regenerate | Title: `Regenerate the edit?` Body: `This discards the current edit and builds a new one from scratch. Guests are not notified either way.` Confirm: `Regenerate` / Cancel: `Cancel` |

Copy rules:
- Never imply guests have seen anything before approval — every pre-approval state must reassure "guests haven't been notified".
- Approve confirm copy MUST name both channels (push + email) and the all-guests fan-out, and state it can't be undone (the notification, not the video).

## Interaction Contract (state machine)

The review surface is a single card on the host event-detail surface (web `pages/host/event-detail.tsx` "Same-day edit" card; mobile `app/(tabs)/event.tsx` `videoCard`), shown only when `event.status === "ended"` and the viewer is the host. It is driven by the video-status response, which gains review/approved states (planner chooses additive status vs `approvedAt`; UI must handle both a `ready_for_review`-style state and an `approved` state distinctly from the old `completed`).

| State | Player | Actions shown | Notes |
|-------|--------|---------------|-------|
| no job / pending | hidden | none | Empty/queued copy; poll active (existing `refetchInterval: 3000` while `pending`/`processing`) |
| processing (compiling) | hidden | none | Spinner + "Compiling…" copy; keep polling |
| ready_for_review | `<video>` / `expo-av` at signed URL | **Approve & notify guests** (primary), **Regenerate edit** (outline) | The core gate. Player visible, both actions enabled |
| regenerating | hidden | none (Regenerate disabled while a new job runs) | "Building a fresh edit…" copy; poll until the new job reaches review |
| approved / delivered | `<video>` (still watchable) | none, or **Download** if already present for the tier | Success treatment; actions removed so it can't be re-sent |
| failed | hidden | **Regenerate edit** (recovery) | Destructive-tinted card + error message |

Action affordances:
- **Approve** opens a confirmation dialog (web: existing `Dialog` from `@/components/ui/dialog`, mirroring `showEndDialog`; mobile: `Alert.alert` confirm, mirroring the existing "End Event" confirm in `event.tsx`). Approve is the only path that triggers `POST /api/events/:eventId/video/approve`.
- **Regenerate** also confirms (lighter consequence — no guest notification) before calling `POST /api/events/:eventId/video/regenerate`, then transitions the card to the regenerating state and keeps polling.
- Buttons show a pending/disabled label while their mutation is in flight (`Approving…`, `Regenerating…`), matching the existing `endEvent.isPending ? "Ending..."` pattern.
- Loading and error of the status query itself reuse existing patterns: `Skeleton` (web) / `ActivityIndicator` (mobile) while loading; a toast on mutation error (`useToast` web / `Alert` mobile), consistent with `handleEndEvent`.

## Assumptions (gaps filled, not from upstream)

These were not pinned by REQUIREMENTS.md / CONTEXT.md / RESEARCH.md and were filled with conventions consistent with the existing codebase. Flag any you want changed.

1. **Status-state labels and copy** above are proposed; CONTEXT.md fixes the *states* (`ready_for_review`, `approved`, etc.) but not the exact wording.
2. **Approve uses a modal confirm on web and `Alert.alert` on mobile** (mirrors the existing End/Delete event confirms) rather than an inline two-tap button. This satisfies the high-consequence confirm requirement with zero new components.
3. **Mobile host review card must switch from the token video-status hook to the authed host hook** so the unapproved review video is never served on the public/token path — this is a correctness requirement implied by CONTEXT.md ("Public/token status must NOT expose the review video before approval"), surfaced here as a UI wiring note.
4. **Regenerate is an `outline`/secondary button**, never accent-colored, to keep Approve as the single primary action.
5. **Approved state hides both actions** (rather than disabling) to prevent re-sending; Download is shown only if the tier already grants it (existing Pro-gating pattern in `video-playback.tsx`).
